# Spec: MVP — 每日 brief（第一周自由探索版）

- 状态：已锁定（仅本周；周末复盘后可改）
- 作者：团队 + Claude
- 日期：2026-04-25
- 关联：[`product.md`](./product.md)

## 目标

第一周内，让团队 8 人**各自独立**完成一份 MVP 实现：每天能产出一篇关于 Agentic Software Engineering 的 brief，发布到一个静态站点上能看。

> 第一周的目标不是产出"产品"，而是收集 **8 份对同一 spec 的不同实践**——下周对比、抽取共识、再做系统化设计。这意味着：本周的"成功"不在于谁做得最好，而在于每人都跑通了端到端、并诚实记下了过程。

## 用户故事

作为团队成员，我打开自己实现的网站，能看到至少一篇符合 [输出契约](#输出契约所有人必须一致) 的 brief；每条要点能点回原文。

## 验收条件

每人交付时自验，逐条勾选：

- [ ] 至少跑出 **1 篇真实数据**生成的 brief（不是手写样例）
- [ ] brief 文件位置和命名符合输出契约
- [ ] brief 章节结构符合输出契约
- [ ] 每条要点都有可点击的原文链接
- [ ] 当日无重要事件时使用兜底文案、**不凑数**
- [ ] 静态站点能本地预览，能从首页点进当天的 brief
- [ ] `experiments/<你的名字>/NOTES.md` 已写
- [ ] 已 push 到本仓库（**仅在 `experiments/<你的名字>/` 子目录下**，不改他人代码）

## 输出契约（**所有人必须一致**）

> 这是本周唯一的硬约束。其他实现细节都自由。
> 之所以要这一条：下周想横向对比 8 份实践，输出形态必须统一，否则只能各说各话。

### 文件位置与命名

- brief 文件放在 `experiments/<你的名字>/briefs/YYYY-MM-DD.md`
- `YYYY-MM-DD` 是 brief **覆盖**的日期（不是生成时间）

### 文件结构（Markdown）

```markdown
# Daily Brief — YYYY-MM-DD

> 一句话摘要：今天最值得关注的方向是 ...
>
> 数据源：N 个 / 已扫条目：M / 入选条目：K

## 最关注的事

### 1. <一句话标题>

<一段话说清是什么、为什么重要，2–4 句>

来源：[<原文标题>](<URL>)

### 2. ...

（最多 3 条；每条必须带原文链接；按重要性排序）

## 值得一看的事

- <一句话> — [<源>](<URL>)
- <一句话> — [<源>](<URL>)

（最多 5 条；每条必须带原文链接）

## 今日观察小结（可选）

<不超过 100 字的元评论；可不写>
```

### 兜底文案（无重要事件时）

依然生成文件，**不要跳过**：

```markdown
# Daily Brief — YYYY-MM-DD

> 今日无重要事件。
>
> 数据源：N 个 / 已扫条目：M / 入选条目：0

## 说明

今日扫描的 N 个源中，没有达到入选标准的事项。明天再来。
```

> "什么算重要"——完全交给每人自己定义。这正是下周对比的核心维度之一。

## 工作目录约定

- 每人在 `experiments/<你的名字>/` 下做所有实现
- 子目录结构、技术栈、依赖管理方式完全自由
- **不要**改 `experiments/` 之外的任何文件——发现 `spec/` 或 `CLAUDE.md` 有问题，先在群里提，由 PM 统一改
- 提交信息建议带前缀：`feat(<name>): ...`，方便后续按人筛历史

## NOTES.md（每人必交）

`experiments/<你的名字>/NOTES.md`，是下周分享会的素材。建议结构：

```markdown
# <你的名字> 的实践笔记

## 技术栈
用了什么语言、库、LLM、静态生成器、托管方式

## 信息源
选了哪些源、为什么；URL 清单

## 流水线设计
fetch → dedup → summarize → render 你怎么拆的；是否引入 skill / agent / 多 agent

## Prompt 思路
核心 prompt 长什么样、迭代了几次、关键决策

## 成本 / 时延
跑一次需要多久、消耗多少 token / 钱（粗估）

## 出乎意料的事
用 Claude Code 协作中的好 / 差，2–3 条

## 如果再来一次
你会做得不一样的地方
```

## 不做什么

- ❌ 用户登录 / 认证
- ❌ 动态后端（DB / API 服务）
- ❌ 邮件 / IM 推送
- ❌ 趋势预测、辣评、深度报告等其他形态
- ❌ **不要追求质量完美**——跑通端到端、产出 1 篇符合契约的 brief 就达标。质量是第二周以后的事
- ❌ 不要在 `experiments/` 之外做架构、目录骨架、共享库——下周再说

## 设计要点 / 约束

- 输出契约是硬约束，其他全部软约束
- LLM 用什么模型自选（用你能调用的，OpenAI / Anthropic / 本地都行）
- 不需要数据库；中间产物用文件系统 / 内存就够
- 不需要 CI；本地能跑、能 push 就够
- 信息源**自选**（建议 5–10 个），但务必在 NOTES.md 写清选了哪些、为什么

## 开放问题

> 有意保留——每人自己回答，下周对比

- 抓取阶段怎么处理失败 / 限流？
- 去重用什么策略？URL 规范化 / 标题相似度 / 语义聚类？
- "重要性"如何判定？人工规则 / LLM 评分 / 混合？
- 如何确保每条要点的原文链接准确（不串源、不幻觉 URL）？
- prompt 一次性出 brief vs. 多步分解，哪个更好？
- 静态站点用什么工具，还是手写 HTML / 直接 Markdown 渲染？
