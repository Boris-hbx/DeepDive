# experiments/Wei — 本地补充规则

> 本文件在项目根 `CLAUDE.md` 基础上叠加生效，根目录规则全部继承，不重复列出。
> 本目录归属：Wilson（wei）

## 开发透明性与可观测

### 工作日志

每次新增需求和代码修改任务结束前，在 `Worklog/wilson-notes.md` 追加一条记录，格式：

```
## YYYY-MM-DD

- 做了什么（一句话总结任务）
- 为什么修改
- 风险和假设
- 如何验证的
```

每次问题修复结束前，在 `Worklog/wilson-notes.md` 追加一条记录，格式：

```
## YYYY-MM-DD

- 做了什么（一句话总结任务）
- 关键判断点：遇到了什么选择，选了哪个，为什么
- 踩坑 / 修复：现象 → 根因 → 修复方法（非显然的才记）
```


不要事后补——每跑通一个阶段当场写两行，细节会随时间流失。
不要记录隐私信息，API keys，password等


### AI 操作可见性

每次 session 开始时，AI 应主动说明本次打算做什么（一句话），遇到判断点停下来问，不要静默推进。

具体要求：
- 读文件前说"先读 X 了解 Y"
- 改代码前说"打算改 X，原因是 Y"
- 遇到 spec 没覆盖的情况，列出选项再问，不要自行决定
- 完成后说"做了什么，下一步建议是什么"，不复述 diff

### 决策痕迹

非显然的技术决策（选了某个方案、放弃某个方案、改了 spec）写入 `Worklog/wilson-notes.md` ，方便本地快速回溯。

### 中间产物落盘

流水线每一步的中间产物（fetch → dedup → rank → summarize）都落盘到 `data/YYYY-MM-DD/` 下对应的 JSON 文件。任何步骤失败只需重跑该步，不从头来。AI 不得跳过落盘步骤做"内存中直通"的优化。

## Wei 目录专属约束

- `Worklog/` 目录只增不删，历史记录不覆盖
- `spec/` 下的文件改动必须先问，得到确认再动（继承根目录规则，此处强调）
- `.env` 不入库，凭证只放 `.env`，不硬编码在代码里
- 引入新 npm 依赖前先确认，说明用途和替代方案

## 本目录技术上下文

- 语言：Node.js ESM（`.mjs`）
- 主入口：`insight-cli.mjs`（CLI）、`server.mjs`（HTTP + 定时任务）
- 核心 lib：`lib/` 下各模块，见 `Worklog/wilson-notes.md` 的架构说明
- 信息源配置：`sources.json`（双领域：软件工程 + 网络安全）
- 报告归档：`reports/YYYY/MM/DD/`，每日 brief：`briefs/YYYY/MM/DD/`
