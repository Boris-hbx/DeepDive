E4M3 最大值推导公式错误
Test Case Bug

错误现象

# test_fp8_quantization.py::TestBlockWiseQuantization::test_e4m3_max_value
e4m3_max = 1.875 * (2 ** 8)  # = 480
assert e4m3_max == 448.0      # FAILED: 480.0 != 448.0

根因分析

Analysis 报告声明："E4M3 最大值约 448" — 结论正确，但未给出推导过程。

Test Case 错误原因：测试自行推导时，错误假设最大尾数为 1.875（二进制 1.111）。
实际上 E4M3fn 格式中，位模式 S_1111_111（指数全1 + 尾数全1）被保留为 NaN， 因此最大指数下的最大尾数只能是 1.110₂ = 1.75。

正确推导：
• E4M3fn: 1 sign + 4 exponent + 3 mantissa, bias = 7
• 最大存储指数 = 15 → 无偏指数 = 8
• 最大指数下最大尾数 = 1.110₂ = 1.75（∵ 尾数=111 + 指数=1111 → NaN）
• Max = 1.75 × 2⁸ = 448
• 验证: torch.finfo(torch.float8_e4m3fn).max == 448.0 ✓

修复代码

# Before (wrong):
e4m3_max = 1.875 * (2 ** 8)  # = 480, NOT 448

# After (correct):
e4m3_max = 1.75 * (2 ** 8)   # = 448 ✓
assert e4m3_max == 448.0
assert torch.finfo(torch.float8_e4m3fn).max == 448.0
结论：Test Case 推导错误。Analysis 报告的 "≈448" 结论正确。

MHA KV Cache 算术计算错误 (2,002,944 vs 1,998,848)
Analysis 报告 Bug

错误现象

# test_mla_math.py::TestKVCacheCalculations::test_mha_kv_cache_per_token
kv_per_token = 2 * 128 * 128 * 61
assert kv_per_token == 2_002_944  # FAILED: 1,998,848 != 2,002,944

根因分析

Analysis 报告原文：
KV_cache_MHA = 2 × 128 × 128 × seq_len × 61 = 2,002,944 × seq_len

实际验算：
2 × 128 × 128 × 61 = 32,768 × 61 = 1,998,848

差异：2,002,944 − 1,998,848 = 4,096（恰好 = 2¹²）

推测原因：报告作者手动计算时可能混入了额外维度（如 qk_rope_head_dim=64 的部分贡献），或单纯的计算笔误。

对结论的影响：
• 报告声称压缩比: 2,002,944 / 35,136 ≈ 57.01x
• 实际压缩比: 1,998,848 / 35,136 ≈ 56.89x
• "约 57x" 的定性结论仍然成立，影响可忽略

修复代码

# Before (copied wrong number from report):
kv_per_token = 2 * n_heads * head_dim * n_layers
assert kv_per_token == 2_002_944

# After (correct arithmetic, step by step):
kv_per_token_per_layer = 2 * n_heads * head_dim
assert kv_per_token_per_layer == 32_768   # 2 × 128 × 128

kv_per_token = kv_per_token_per_layer * n_layers
assert kv_per_token == 1_998_848           # 32,768 × 61
结论：Analysis 报告算术错误。2 × 128 × 128 × 61 = 1,998,848，非 2,002,944。

Sigmoid 测试因高维点积数值溢出失败
Test Case Bug

错误现象

# test_moe_routing.py::TestSigmoidScoring::test_sigmoid_not_softmax
x = torch.randn(B, 7168)           # 7168 维随机向量
weight = torch.randn(256, 7168)     # 256 个专家权重
logits = torch.matmul(x, weight.T)  # 点积 std ≈ √7168 ≈ 84.7
scores = logits.sigmoid()

assert (scores > 0).all()  # FAILED: sigmoid(-90) → 0.0 in float32

根因分析

Analysis 报告声明：score_func = sigmoid — 完全正确。

Test Case 错误原因：测试用 7168 维标准正态随机向量做点积。
两个 d 维标准正态向量的点积服从 N(0, d)，当 d=7168 时标准差 ≈ 84.7。
这意味着 logit 值经常超过 ±80，而 sigmoid(x) 在 x < −88 时 float32 精度下精确等于 0.0。

关键点：这不是 sigmoid 函数本身的问题。实际模型中 gate 权重经过训练，logit 量级可控。 测试的目的是验证 "sigmoid 分数不归一化" 这一性质，不需要模拟真实维度。

修复代码

# Before (high-dim dot product causes sigmoid saturation):
x = torch.randn(B, 7168)
weight = torch.randn(256, 7168)
logits = torch.matmul(x, weight.T)  # |logits| can reach 200+
scores = logits.sigmoid()
assert (scores > 0).all()            # fails: saturated to 0.0

# After (controlled magnitude, focused on the property being tested):
logits = torch.randn(B, n_experts) * 2.0  # moderate range
scores = logits.sigmoid()
assert (scores >= 0).all()
assert (scores <= 1).all()

# Key: sigmoid scores are NOT normalized (unlike softmax)
assert not scores.sum(dim=-1).allclose(torch.ones(B))
softmax_scores = logits.softmax(dim=-1)
assert softmax_scores.sum(dim=-1).allclose(torch.ones(B))
结论：Test Case 输入构造不当（数值溢出）。Analysis 报告关于 sigmoid 的声明正确。
