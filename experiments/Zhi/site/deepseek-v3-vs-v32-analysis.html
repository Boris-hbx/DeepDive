一、核心发现摘要

V3.2 的最大变化是引入了 Indexer（稀疏注意力索引机制）——这是一个全新的模块，通过 top-k 选择机制让注意力只关注最相关的 token，而非全部历史 token。这是对长序列推理效率的根本性优化。

其他关键变化：全面 FP8 量化（含 KV cache）、残差连接重构、MoE 精度提升。

二、架构变化

2.1 新增模块：Indexer（稀疏注意力）

影响：高

V3.2 新增了一个完整的 Indexer 类（约 70 行），这是 V3 中完全不存在的：

维度V3V3.2
注意力范围全序列（O(n²)）Top-2048 token（稀疏）
索引头数无64 heads, dim=128
索引方式无Hadamard 旋转 + FP8 量化 + top-k

工作原理：

  1. 用独立的 Q/K 投影计算索引分数（复用 q_lora_rank）
  2. 对 Q 和 K 做 Hadamard 变换（rotate_activation），量化为 FP8
  3. 通过新的 fp8_index kernel 高效计算索引分数
  4. 选出 top-2048 个 token 的索引，生成稀疏 mask
  5. 注意力计算只在这 2048 个位置上进行

意义：将长序列推理的注意力复杂度从 O(n) 降为 O(2048)（固定常数），对超长上下文场景（如 128K+ token）的推理速度有质的提升。

2.2 残差连接重构

影响：中
V3V3.2
x = x + self.attn(self.attn_norm(x), ...)
x = x + self.ffn(self.ffn_norm(x))		 x, residual = self.attn_norm(x, residual)
x = self.attn(x, ...)
x, residual = self.ffn_norm(x, residual)
x = self.ffn(x)

V3.2 将残差加法融入 RMSNorm 内部，residual 作为独立张量在层间传递。好处：

  • 避免 FP8 精度下残差累积误差
  • RMSNorm 内部在 float32 下做加法，精度更高
  • 减少一次显式加法操作的显存读写

2.3 移除 naive attention 路径

影响：低

V3 保留了 attn_impl: "naive" | "absorb" 两种实现。V3.2 移除了 naive 路径，只保留 absorb（MLA 吸收式注意力），说明 absorb 已经是唯一的生产路径。

三、推理效率变化

3.1 FP8 全面化

影响：高
组件V3V3.2
权重存储FP8（可选 bf16 dequant）FP8 only
KV Cachebf16/fp16FP8 量化
Indexer Cache无FP8
GEMM 实现bf16 或 fp8 可选fp8 only（移除 bf16 路径）

KV Cache FP8 量化是关键优化：

# V3.2 新增：KV cache 存储前先量化为 FP8，读取时反量化
kv_fp8, kv_scale = act_quant(kv, block_size, self.scale_fmt)
kv = (kv_fp8.view(-1, block_size).float() * kv_scale.view(-1, 1)).to(kv.dtype).view_as(kv)
self.kv_cache[:bsz, start_pos:end_pos] = kv

效果：KV cache 显存占用减半（bf16 → fp8），直接提升可服务的最大序列长度和批量大小。

3.2 稀疏注意力的效率收益

Indexer 的 top-2048 机制意味着：

  • Prefill 阶段：注意力从 O(n²) 降为 O(n × 2048)，对 128K 序列约 62x 加速
  • Decode 阶段：每步只需关注 2048 个历史 token，而非全部
  • 代价：额外的 Indexer 前向计算（64 heads × 128 dim），但远小于全注意力开销

3.3 计算精度优化

V3.2 在多处将中间计算提升到 float32：

  • MLP 激活：F.silu(self.w1(x).float()) * self.w3(x).float()（V3 在 bf16 下计算）
  • MoE 路由分数：linear(x.float(), self.weight.float())
  • MoE 累加：y = torch.zeros_like(x, dtype=torch.float32)
  • LM head：self.head(h[:, -1].float())

这看似矛盾（FP8 存储 + FP32 计算），实际是精心设计：存储用低精度省显存，计算用高精度保质量。

3.4 权重反量化缓存

# V3: 每次 decode 都重新反量化
wkv_b = weight_dequant(self.wkv_b.weight, self.wkv_b.scale, block_size)

# V3.2: 首次反量化后缓存，后续复用
if self.dequant_wkv_b is None and self.wkv_b.scale is not None:
    self.dequant_wkv_b = weight_dequant(self.wkv_b.weight, self.wkv_b.scale)
wkv_b = self.wkv_b.weight if self.dequant_wkv_b is None else self.dequant_wkv_b

Decode 阶段每步都要用 wkv_b，缓存反量化结果避免重复计算。

四、代码实现变化

4.1 新增 Kernel：fp8_index

V3 的 kernel.py 只有 act_quant、weight_dequant、fp8_gemm。V3.2 新增 fp8_index——专门为 Indexer 设计的 FP8 索引计算 kernel，用于高效计算稀疏注意力分数。

4.2 RoPE 支持 non-interleaved 模式

# V3: 只有 interleaved
def apply_rotary_emb(x, freqs_cis):

# V3.2: 支持两种模式
def apply_rotary_emb(x, freqs_cis, interleaved=True):

Indexer 使用 non-interleaved RoPE（apply_rotary_emb(q_pe, freqs_cis, False)），与主注意力的 interleaved 模式区分开。

4.3 新增 LayerNorm（带 bias）

V3 只有 RMSNorm。V3.2 新增标准 LayerNorm（带 weight + bias），专门用于 Indexer 的 K 归一化（self.k_norm = LayerNorm(self.head_dim)）。

4.4 MoE 共享专家的 reduce 行为变化

# V3: shared_experts 独立计算，最后加到 routed 结果上
z = self.shared_experts(x)
return (y + z).view(shape)

# V3.2: shared_experts 不做 all-reduce，直接累加到 float32 结果中
self.shared_experts = MLP(args.dim, ..., reduce_output=False)
y += self.shared_experts(x)
return y.type_as(x).view(shape)

减少一次 all-reduce 通信，将共享专家的输出直接在本地累加。

六、关键洞察

  1. Indexer 是 V3.2 的核心创新——它本质上是一个"注意力路由器"，用轻量级计算决定哪些 token 值得关注，类似 MoE 对 FFN 做的事情（选择性激活），Indexer 对注意力做了同样的事。
  2. FP8 从"可选"变为"唯一"——V3 还保留 bf16 回退路径，V3.2 完全移除。说明 FP8 推理已经成熟到可以作为默认方案。
  3. "低精度存储 + 高精度计算"的设计哲学——存储（权重、KV cache）用 FP8 省显存，但关键计算路径（MLP 激活、路由分数、logits）提升到 FP32。这是工程上的精细权衡。
  4. 模型规模不变，效率大幅提升——671B/37B 的参数配置完全不变，所有优化都在推理效率层面，说明 V3.2 是一次纯工程优化迭代。
