项目目标

构建一个 AI/ML 领域的洞察分析系统，包含两个核心能力：

  • Daily Brief — 每日自动抓取、过滤、摘要 AI 领域动态，生成可读的技术简报
  • Deep Dive — 对特定技术主题（如 DeepSeek 架构演进）进行深度分析，产出结构化的技术报告，并通过可执行测试验证报告中的技术声明

五、质量验证：Bug 发现与修复

50 个测试用例中发现 3 个 bug，全部修复。这个过程本身验证了"用测试验证分析报告"方法论的有效性。

#Bug归属根因
1 E4M3 最大值计算错误（480 vs 448） Test Bug 未考虑 E4M3fn 的 NaN 编码保留位，max mantissa 是 1.75 而非 1.875
2 MHA KV Cache 计算偏差（2,002,944 vs 1,998,848） Report Bug 报告中 2×128×128×61 的乘法结果写错，差值 = 4096 = 2¹²
3 Sigmoid 饱和导致测试误判 Test Bug 7168 维随机向量点积 std≈84.7，sigmoid 饱和到 0.0/1.0
关键洞察

Bug #2 证明了这套方法论的价值：分析报告中一个看似无害的算术错误（影响压缩比从 57.0x 变为 56.9x），被测试精确捕获。如果没有可执行验证，这类错误会一直留在文档中误导读者。

六、经验与教训

做得好的

高效

AI 辅助开发速度惊人 — 从零到完整流水线跑通只用了 20 分钟。关键是先想清楚架构（即使只是 5 行描述），再让 AI 实现。

方法论

"分析 → 验证 → 修复"闭环有效 — 用 pytest 验证技术文档的声明，把主观的"这个分析对不对"变成客观的"50 个测试全绿"。这个方法可以复用到任何技术分析场景。

简洁

不过度设计 — 没有引入 agent 框架、没有用前端框架、没有搭数据库。纯 Python + 函数调用串联，够用就行。当前规模（5 源、日更）不需要更复杂的架构。

踩过的坑

工具链

Claude API 的 ThinkingBlock 陷阱 — Opus 4.6 默认启用 extended thinking，返回的第一个 content block 不是 TextBlock。必须过滤 [b for b in content if b.type == "text"]。

工具链

大文件写入不稳定 — 超过 150 行或含大量 HTML 特殊字符的文件，Write 工具会序列化失败。解决方案：改用 Bash heredoc 写入。

测试设计

测试数据的量级要匹配真实场景 — Bug #3 就是因为测试用了 7168 维随机向量（匹配真实模型维度），但没考虑高维点积的统计特性会导致 sigmoid 饱和。测试数据需要同时考虑"维度正确"和"数值合理"。

方法论沉淀

方法论

技术分析报告的可验证性标准 — 一份好的技术分析报告，其中的每个数学声明、参数值、代码逻辑都应该能被独立验证。如果一个声明无法写成测试，要么是声明太模糊，要么是理解不够深。

方法论

Bug 归属分析的价值 — 区分"测试写错了"和"报告写错了"很重要。前者说明验证者对技术细节理解不够，后者说明分析本身有误。两种情况的修复路径完全不同。

总结

3 天时间，从零构建了一个完整的 AI 洞察分析系统：日报流水线稳定运行 3 期，深度分析报告经过 50 个测试用例验证，发现并修复了 3 个 bug。

最大的收获不是代码本身，而是验证了一套方法论：技术分析的质量可以用可执行测试来度量。这把"我觉得分析得对"变成了"50 个测试证明分析是对的"，是从主观判断到客观验证的跨越。
