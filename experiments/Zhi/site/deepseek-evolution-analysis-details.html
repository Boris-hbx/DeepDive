一、MLA（Multi-head Latent Attention）机制详解

核心思想：MLA 通过低秩投影将 KV cache 从 n_heads × head_dim 压缩到一个共享的低秩潜在向量 kv_lora_rank，在推理时只需缓存这个压缩向量，大幅减少显存占用。

1.1 完整超参数（V3 671B 实际配置）

参数值含义
dim7168模型隐藏维度
n_heads128注意力头数
q_lora_rank1536Query 低秩压缩维度
kv_lora_rank512KV 低秩压缩维度（核心！）
qk_nope_head_dim128不带位置编码的 QK 维度
qk_rope_head_dim64带 RoPE 的 QK 维度
v_head_dim128Value 头维度
n_layers61Transformer 层数
n_dense_layers3前 3 层使用 Dense MLP

1.2 数学公式

数学推导

标准 MHA 的 KV cache 大小：

KV_cache_MHA = 2 × n_heads × head_dim × seq_len × n_layers
= 2 × 128 × 128 × seq_len × 61 = 2,002,944 × seq_len（每 token 约 2M 参数）

MLA 的 KV cache 大小：

KV_cache_MLA = (kv_lora_rank + qk_rope_head_dim) × seq_len × n_layers
= (512 + 64) × seq_len × 61 = 35,136 × seq_len（每 token 约 35K 参数）

压缩比： 2,002,944 / 35,136 ≈ 57x

关键洞察：MLA 的 KV cache 只需存储两样东西：(1) 压缩后的 KV 潜在向量 c_kv（512维），(2) 解耦的 RoPE key k_pe（64维）。总共 576 维 vs 标准 MHA 的 32,768 维（128头×256维）。

scores *= softmax_scale

# 输出也在压缩空间计算，最后再展开
output = einsum("bsht,btc->bshc", attn_weights, kv_cache)     # [B,S,H,512]
output = einsum("bshc,hdc->bshd", output, wkv_b[:, -128:])    # 展开 V

1.4 RoPE 解耦设计的精妙之处

MLA 面临一个核心矛盾：RoPE 是位置相关的，但 KV 压缩需要位置无关的表示。DeepSeek 的解决方案：

  1. 将 QK 分为两部分：128 维不带位置编码（nope），64 维带 RoPE（rope）
  2. KV 压缩只作用于 nope 部分：512 维的 c_kv 通过 wkv_b 展开为 128 维的 k_nope
  3. RoPE key 单独存储：64 维的 k_pe 不经过压缩，直接缓存
  4. 注意力分数 = nope 部分 + rope 部分：两部分独立计算后相加

这样既保留了位置信息，又实现了 KV 的高效压缩。代价是多存 64 维的 k_pe，但相比节省的 KV cache 微不足道。

二、Indexer 稀疏注意力机制（V3.2 首创）

核心思想：V3.2 引入 Indexer 模块，在注意力计算前先用轻量级打分器选出最相关的 top-k token（默认 2048），只对这些 token 做完整注意力。这是 DeepSeek 首次将"选择性计算"从 FFN 层（MoE）推广到注意力层。

2.2 Indexer 工作流程

代码分析 
class Indexer:
    # 输入：x (当前 token 表示), qr (query 的低秩压缩)

    # Step 1: 生成 Indexer 专用的 Q 和 K
    q = wq_b(qr)                          # 复用主注意力的 q_lora_rank 输出
    q = reshape(q, [B, S, 64, 128])       # 64 个 indexer 头
    q_pe, q_nope = split(q, [64, 64])     # RoPE 部分 + 非 RoPE 部分
    q_pe = apply_rotary_emb(q_pe)         # 注意：indexer 的 RoPE 不是 interleaved

    k = wk(x)                             # 独立的 K 投影 (7168 → 128)
    k = LayerNorm(k)                      # 注意：用 LayerNorm 而非 RMSNorm
    k_pe, k_nope = split(k, [64, 64])
    k_pe = apply_rotary_emb(k_pe)

    # Step 2: Hadamard 旋转（关键！）
    q = hadamard_transform(q, scale=128^-0.5)
    k = hadamard_transform(k, scale=128^-0.5)

    # Step 3: FP8 量化后计算相似度
    q_fp8, q_scale = act_quant(q, block_size=128)
    k_fp8, k_scale = act_quant(k, block_size=128)

    # Step 4: 加权打分
    weights = weights_proj(x.float()) * n_heads^-0.5  # [B, S, 64]
    weights = weights * q_scale * softmax_scale

    # Step 5: FP8 索引计算（自定义 kernel）
    index_score = fp8_index(q_fp8, weights, k_cache, k_scale_cache)

    # Step 6: Top-k 选择
    topk_indices = index_score.topk(min(2048, seq_len))[1]
    return topk_indices

2.3 关键设计决策

为什么用 Hadamard 变换？

Hadamard 变换是一种正交变换，将向量的能量均匀分散到所有维度。这对 FP8 量化至关重要：

  • 原始向量可能有少数维度值很大（outlier），FP8 量化会损失精度
  • Hadamard 变换后，所有维度的值域更均匀，FP8 量化误差更小
  • 计算复杂度 O(d log d)，开销极小

为什么 Indexer 用 FP8 而非 BF16？

Indexer 的目的是快速粗筛，不需要精确的注意力分数。FP8 的精度足够区分"相关"和"不相关"的 token，同时：

  • K cache 存储量减半（FP8 vs BF16）
  • 矩阵乘法吞吐量翻倍（FP8 GEMM）
  • Indexer 本身的计算开销降到最低

Indexer 如何与主注意力配合？

# 在 MLA.forward() 中：
topk_indices = self.indexer(x, qr, start_pos, freqs_cis, mask)

# 构造稀疏 mask：只允许 attend 到 top-k 位置
index_mask = full((B, S, seq_len), -inf)
index_mask.scatter_(-1, topk_indices, 0)  # top-k 位置设为 0（允许）

# 叠加到注意力分数上
scores += index_mask  # 非 top-k 位置被 mask 掉（-inf → softmax 后为 0）

2.4 V3 vs V3.2 残差连接的变化

实现差异

V3.2 对残差连接做了重要修改——将残差加法融入 RMSNorm：

# V3 的 Block.forward()：
x = x + self.attn(self.attn_norm(x), ...)    # 标准 Pre-Norm
x = x + self.ffn(self.ffn_norm(x))

# V3.2 的 Block.forward()：
x, residual = self.attn_norm(x, residual)     # Norm 内部做残差加法（float32）
x = self.attn(x, ...)
x, residual = self.ffn_norm(x, residual)      # 同上
x = self.ffn(x)

V3.2 的 RMSNorm 实现：

def forward(self, x, residual=None):
    if residual is None:
        x = x.float()  # 提升到 float32
        ...
    else:
        x = residual = x.float() + residual.float()  # 残差加法在 float32 下进行！
        var = x.pow(2).mean(-1, keepdim=True)
        x = x * rsqrt(var + eps)
        return (self.weight * x).to(dtype), residual.to(dtype)
为什么这很重要？在 FP8 推理中，残差流是精度的"生命线"。如果残差加法在低精度下进行，误差会逐层累积。V3.2 将残差加法提升到 float32，确保信号在 61 层传播后不会退化。这是 V4 "流形约束超连接"的工程前身。

三、V4 混合稀疏注意力（CSA + HCA）

核心架构：V4 的每一层注意力由三个组件混合构成：SWA（滑动窗口注意力，128 token）+ 两种压缩注意力之一：C4（4:1 压缩 + top-512 稀疏）或 C128（128:1 压缩 + 全量密集）。不同层使用不同的组合。

3.2 CSA（Compressed Sparse Attention）详解

架构核心

CSA 是 V3.2 Indexer 的系统化升级版：

# CSA 工作流程（C4 层）

# Step 1: 压缩 KV（4:1 比例）
# 每 4 个连续 token 的 KV 通过 softmax 加权平均压缩为 1 个
compressed_kv = compress(raw_kv, ratio=4)  # 1M tokens → 250K compressed positions

# Step 2: Lightning Indexer 打分
# 用轻量级 GEMM 计算 query 与所有压缩位置的相关性
index_scores = indexer_gemm(query, compressed_keys)  # [B, H, 250K]

# Step 3: Top-k 选择（Lightning TopK）
# 从 250K 个压缩位置中选出 top-512
top_indices = lightning_topk(index_scores, k=512)  # 约 15μs

# Step 4: 只对选中的 512 个压缩位置做完整注意力
selected_kv = gather(compressed_kv, top_indices)
output_c4 = full_attention(query, selected_kv)  # O(512) 而非 O(250K)

Flash Compressor：IO 感知的精确压缩

压缩过程本身是计算密集的（需要 softmax 加权平均），DeepSeek 设计了 Flash Compressor 将 5 次 HBM 访问融合为 1 次：

# 朴素实现（5 次 HBM 读写）：
scores = linear(tokens)           # HBM read + write
weights = softmax(scores)         # HBM read + write
compressed = weighted_avg(kv, weights)  # HBM read + write

# Flash Compressor（1 次 on-chip pass）：
# 所有操作在 SRAM 中完成，softmax 保持 warp-local（C4）
# 或 single CTA-wide reduction（C128）
compressed = flash_compress(tokens, kv)  # 2 次 HBM（读入 + 写出）

3.3 HCA（Heavily Compressed Attention）详解

HCA 是 CSA 的互补：用极高压缩比换取全局覆盖。

# HCA 工作流程（C128 层）

# Step 1: 极度压缩（128:1）
# 每 128 个 token 压缩为 1 个摘要 token
summary_kv = compress(raw_kv, ratio=128)  # 1M tokens → ~8K summaries

# Step 2: 全量密集注意力（无需 top-k 选择）
# 因为只有 ~8K 个位置，全量计算也很快
output_c128 = full_attention(query, summary_kv)  # O(8K)

3.4 层间混合策略

V4 的 61 层 Transformer 中，每层使用以下组合之一：

  • SWA + C4：局部精确 + 全局稀疏（大部分层）
  • SWA + C128：局部精确 + 全局摘要（部分层）

这种设计的直觉：

  • SWA 保证最近上下文的精确性（128 token 窗口，无信息损失）
  • C4 提供全局的细粒度检索能力（选择性关注远处的关键 token）
  • C128 提供全局的粗粒度语义理解（整体文档结构、主题）

3.6 HiSparse：CPU 扩展 KV Cache

C4 层的一个额外优化：由于每步 decode 只访问 top-512 个压缩位置（250K 中的 0.2%），绝大部分 C4 KV cache 在任何时刻都是"不活跃"的。HiSparse 将不活跃的 C4 KV 卸载到 CPU 内存：

  • GPU 只保留小的 working set buffer
  • CPU pinned memory 存储完整的 C4 KV cache
  • 每步根据 top-k 结果从 CPU 拉取需要的 KV
  • 效果：长上下文服务吞吐量提升 3x

四、流形约束超连接（Manifold-Constrained Hyper-Connections）

核心思想：mHC 是标准残差连接 x + f(x) 的泛化。它将残差流替换为多个并行分支的加权混合，权重通过 Sinkhorn 归一化的 GEMM 动态计算。这改善了深层网络中的梯度流动和表示质量。

4.1 从标准残差到 mHC 的演进

# 标准残差连接（V3）
output = x + attention(norm(x))

# 融合残差 Norm（V3.2）—— 工程优化
# 残差加法在 float32 下进行，融入 Norm 层
x_normed, residual = RMSNorm(x, residual)  # residual 在 float32 下累加
output = attention(x_normed)

# 流形约束超连接（V4）—— 理论化
# 多分支加权混合，权重动态计算
branches = [branch_i(x) for i in range(hc_mult)]  # hc_mult 个并行分支
mix_weights = sinkhorn(gemm(x))                     # 动态混合权重
output = sum(w_i * branch_i for w_i, branch_i in zip(mix_weights, branches))

4.2 mHC 的实现细节

实现分析

基于 SGLang/LMSYS 的实现描述：

  • hc_mult 参数：控制并行分支数量
  • Sinkhorn 归一化：对 GEMM 输出做 Sinkhorn 迭代，产生双随机矩阵作为混合权重
  • Split-K kernel：在小 batch decode 时，K 维度跨 CTA 分割以提升 GPU 利用率
  • 融合 kernel：mhc_pre_big_fuse_tilelang 将 RMSNorm + Sinkhorn + 残差混合融合为一个 kernel

4.3 为什么 mHC 比标准残差更好？

标准残差的问题：

  • 所有层共享同一个残差流，信息只能"加法叠加"
  • 深层网络中，早期层的信号可能被后期层的输出淹没
  • 梯度在反向传播时可能出现方向冲突

mHC 的优势：

  • 多分支允许不同类型的信息走不同的"通道"
  • 动态权重让模型根据输入内容决定如何混合
  • "流形约束"确保表示不会偏离有效的表示空间
  • Sinkhorn 归一化保证权重的稳定性（双随机矩阵性质）

V3.2 → V4 的演进逻辑：

V3.2 的"融合残差 Norm"解决了 FP8 下残差精度的工程问题。V4 的 mHC 将这个思路理论化：不仅保证精度，还通过多分支混合提升表示能力。这是典型的"先工程验证，再理论升级"的 DeepSeek 方法论。

五、DeepSeekMoE 与无辅助损失负载均衡

核心创新：V3 首创"无辅助损失"的负载均衡策略——不通过额外的 loss 项来鼓励均衡，而是直接在 gate 的打分上加一个可学习的 bias，在推理时动态调整路由。

5.2 Gate 路由机制（代码级）

代码分析 
class Gate:
    def forward(self, x):
        # Step 1: 计算原始分数
        scores = linear(x, self.weight)           # [B*S, 256]
        scores = scores.sigmoid()                  # sigmoid 打分
        original_scores = scores                   # 保存原始分数（用于最终权重）

        # Step 2: 加 bias（无辅助损失负载均衡的核心！）
        if self.bias is not None:
            scores = scores + self.bias            # [256] 的可学习 bias

        # Step 3: 分组路由（256 专家分为 8 组，每组 32 个）
        scores = scores.view(B*S, 8, 32)          # 分组
        # 每组取 top-2 的和作为组分数
        group_scores = scores.topk(2, dim=-1)[0].sum(dim=-1)  # [B*S, 8]
        # 选出 top-4 组
        top_groups = group_scores.topk(4, dim=-1)[1]          # [B*S, 4]
        # mask 掉未选中的组
        mask = ones(B*S, 8).scatter_(1, top_groups, False)
        scores = scores.masked_fill_(mask.unsqueeze(-1), -inf)
        scores = scores.flatten(1)                             # [B*S, 256]

        # Step 4: 从选中的组中选 top-8 专家
        indices = scores.topk(8, dim=-1)[1]       # [B*S, 8]

        # Step 5: 用原始分数（不含 bias）计算最终权重
        weights = original_scores.gather(1, indices)
        weights /= weights.sum(dim=-1, keepdim=True)  # 归一化
        weights *= 2.5                                 # route_scale

        return weights, indices

5.3 无辅助损失负载均衡的精妙之处

传统方法的问题：

  • 辅助损失（auxiliary loss）会引入"干扰梯度"——模型被迫在"学好任务"和"均衡负载"之间权衡
  • 辅助损失的系数难以调节：太大影响模型质量，太小无法均衡

DeepSeek 的解法：

  • bias 只影响路由决策（哪些专家被选中），不影响最终权重（选中后的加权系数）
  • 最终权重用 original_scores（不含 bias）计算，保证模型输出质量
  • bias 在训练中通过简单的启发式更新：过载的专家 bias 减小，欠载的增大
  • 这完全消除了辅助损失对模型梯度的干扰

代码中的判断条件：

# 只有 dim == 7168（即 671B 模型）才启用 bias
self.bias = Parameter(empty(256, dtype=float32)) if self.dim == 7168 else None

这说明 bias 是在大规模训练中才需要的——小模型的专家数少，自然均衡。

5.4 分组路由的设计意图

256 个专家分为 8 组（每组 32 个），每 token 先选 4 组，再从中选 8 个专家。这种两级路由的好处：

  • 通信效率：在分布式训练中，同组专家通常在同一节点。先选组 = 先选节点，减少跨节点通信
  • 多样性：从 4 个不同组中选专家，避免所有 token 都涌向同一组
  • 计算效率：先粗筛（8 组 → 4 组），再细选（128 个候选 → 8 个），减少 top-k 的计算量

六、FP8 混合精度训练体系

核心贡献：DeepSeek V3 是首个在生产规模（671B 参数）成功使用 FP8 训练的模型。关键在于精心设计哪些计算用 FP8、哪些保持高精度，以及 block-wise 量化策略。

6.1 精度分配策略

组件精度原因
GEMM（前向/反向）FP8 (E4M3)计算密集，FP8 吞吐量翻倍
残差流BF16 → Float32（V3.2+）精度生命线
Softmax / RMSNormFloat32数值敏感操作
Gate 路由Float32路由决策影响全局
Indexer K cache（V3.2）FP8粗筛不需要高精度
MoE 专家权重（V4）FP4 (MXFP4)进一步压缩

6.2 Block-wise 量化（代码级）

代码分析 
# 量化参数
block_size = 128  # 每 128 个元素共享一个 scale

def act_quant(x, block_size=128, scale_fmt="ue8m0"):
    """将激活值量化为 FP8"""
    x_blocks = x.view(-1, block_size)
    # 每个 block 取 abs max 作为 scale
    scale = x_blocks.abs().amax(dim=-1) / 448.0  # E4M3 最大值约 448
    x_fp8 = (x_blocks / scale.unsqueeze(-1)).to(torch.float8_e4m3fn)
    return x_fp8.view_as(x), scale

# 权重量化（离线完成）
# scale 是 2D 的：[ceil(out/128), ceil(in/128)]
weight_scale = Parameter(empty(
    (out_features + 127) // 128,
    (in_features + 127) // 128,
    dtype=float32
))

6.3 V3.2 的 scale_fmt: "ue8m0"

V3.2 配置中的 "scale_fmt": "ue8m0" 是 NVIDIA MXFP 格式：

  • ue8m0：unsigned 8-bit exponent, 0 mantissa = 纯指数 scale（2 的幂次）
  • 每个 scale 只用 8 bit（vs 标准 float32 的 32 bit），4x 压缩
  • 反量化只需位移操作，无需乘法

6.4 V4 的 FP4 专家权重

V4 将 MoE 专家权重从 FP8 进一步压缩到 MXFP4：

  • 每个专家权重只占 4 bit，存储量再减半
  • 使用 Blackwell GPU 的原生 MXFP4 支持（tcgen05.mma 指令）
  • SGLang 修改了 TRTLLM-Gen 的 fused MoE kernel 以支持 MXFP8 激活 × MXFP4 权重
  • 只有 MoE 专家用 FP4，共享专家和注意力层仍用 FP8+

七、训练方法论：MTP、蒸馏与 Muon

7.1 Multi-Token Prediction (MTP)

V3 的 MTP 训练目标：除了预测下一个 token，同时预测未来 2-3 个 token。这提供了更丰富的训练信号。

V4 的 MTP 推理加速：V4 附带一个单层 MTP head（独立训练的 DSv4 decoder 层），用于推测解码（speculative decoding）：

  • MTP head 只做 SWA 注意力（无压缩器、无 Indexer），计算极轻
  • 输入 = 上一步的隐藏状态（h_proj）+ 下一个 token 的 embedding（e_proj）
  • 预测的 token 作为"推测"，如果主模型验证通过则直接接受
  • 配合 ShadowRadix 的 ring buffer 扩展（C4: 8→16, C128: 128→256），EAGLE 推测解码开箱即用

7.2 知识蒸馏（V3 → V3.2 → V4）

DeepSeek 的蒸馏策略是其"低成本训练"的核心：

模型蒸馏来源方法
DeepSeek-R1-DistillR1 (671B)用 R1 生成的 CoT 数据微调小模型
V3.2 (推测)V3 + R1在 V3 基础上用 R1 数据继续训练
V4 Flash (284B)V4 Pro (1.6T)从大模型蒸馏到小模型

7.3 Muon 优化器（V4 新增）

V4 据报道使用了 Muon 优化器（来自 Moonlight 项目）：

  • 核心思想：对梯度做正交化处理（类似 Nesterov 动量 + 正交投影）
  • 优势：比 AdamW 更快收敛，尤其在大 batch size 下
  • 与 FP8 的协同：Muon 的梯度正交化可能有助于缓解低精度训练的梯度噪声
  • 训练效率：V4 的 1.6T 参数在合理的训练预算内完成，Muon 是关键因素之一

7.4 训练基础设施

V3 论文披露的训练细节：

  • 集群：2048 × H800 GPU
  • 训练时间：约 2 个月
  • 训练成本：约 $5.5M（远低于同规模模型）
  • 通信优化：DualPipe（计算和通信重叠）、自研 all-to-all 通信库
  • 并行策略：DP + TP + EP + PP（V4 新增 CP 用于长上下文）

8.3 关键代码差异

RMSNorm 的变化

# V3
def forward(self, x):
    x = x.float()
    var = x.pow(2).mean(-1, keepdim=True)
    x = x * torch.rsqrt(var + self.eps)
    return (self.weight * x).to(dtype)

# V3.2（新增 residual 参数）
def forward(self, x, residual=None):
    if residual is None:
        x = x.float()
        var = x.pow(2).mean(-1, keepdim=True)
        x = x * torch.rsqrt(var + self.eps)
        return (self.weight * x).to(dtype)
    else:
        x = residual = x.float() + residual.float()  # float32 残差加法！
        var = x.pow(2).mean(-1, keepdim=True)
        x = x * torch.rsqrt(var + self.eps)
        return (self.weight * x).to(dtype), residual.to(dtype)

Block 的变化

# V3
def forward(self, x, start_pos, freqs_cis, mask):
    x = x + self.attn(self.attn_norm(x), start_pos, freqs_cis, mask)
    x = x + self.ffn(self.ffn_norm(x))
    return x

# V3.2
def forward(self, x, residual, start_pos, freqs_cis, mask):
    if residual is None:
        x, residual = self.attn_norm(x), x
    else:
        x, residual = self.attn_norm(x, residual)
    x = self.attn(x, start_pos, freqs_cis, mask)
    x, residual = self.ffn_norm(x, residual)
    x = self.ffn(x)
    return x, residual
