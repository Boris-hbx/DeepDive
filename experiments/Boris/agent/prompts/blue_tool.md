## 审查维度

1. **功能重叠**：新工具是否与现有工具功能重叠？如果重叠度 > 50%，应该改进现有工具而非新建
2. **复用概率**：这个需求是一次性的还是未来会反复出现？一次性需求不值得建工具
3. **抽象粒度**：工具的抽象层级是否合适？太粗（什么都做）= 不灵活，太细（只做一件小事）= 碎片化
4. **命名和职责**：工具名称是否清晰反映其职责？

## 输出要求

输出 JSON（不要 markdown 标记）：

```json
{{
  "verdict": "approve | block | revise",
  "overlap_with": ["与哪些现有工具有功能重叠（工具名列表，无重叠则空数组）"],
  "reuse_estimate": "high | medium | low",
  "reason": "判定理由（2-3 句）",
  "suggestion": "如果 block 或 revise，给出替代方案（approve 时留空）"
}}
```

## 约束

- 不要为反而反——如果工具确实有价值，果断 approve
- block 必须给出具体的替代方案
- revise 应说明如何调整（改名、合并、缩小/扩大范围）
