02核心理念：Agent 不只用工具，还能创造工具

传统 Agent
固定工具集 → 遇到新任务类型时无能为力
人工编写每个工具 → 扩展慢
工具之间无协同演进
Self-Evolving Agent
动态工具库 → 遇到新模式自动生成工具
反思循环 → 识别可复用操作并沉淀
工具积累效应 → 越用越强

设计原则

  • 单 Agent、多 Workflow — 按任务类型自动选择工作流
  • 蓝军对抗 — Blue Agent 审查报告和工具创建决策
  • 轻量版本管理 — archive/ 目录归档旧版，不用 git-for-tools
  • 量化评估 — benchmark 体系追踪每次运行的质量指标
  • CLI 驱动 — python -m agent "任务描述" 即可运行

06反思与自进化机制

创建新工具的条件（满足任一）
• 同样的操作出现了 2 次以上
• 当前步骤花了大量代码但操作可复用
• 发现通用的数据获取 / 文本处理 / API 模式
改进工具的条件
• 现有工具在当前任务中暴露了不足
• 新需求超出工具当前能力范围
• 旧版自动归档到 archive/
反思输出格式 (JSON) — 含 workflow 评估
{ "step_assessment": "成功抓取了 HN 数据，但过程中手动写了 HTTP 请求代码", "need_new_tool": true, "tool_spec": { "name": "fetch_hn", "description": "Fetch top stories from HN", "language": "python", "args": "keyword: str, limit: int = 10" }, "reasoning": "抓取 HN 是高频操作，应该抽成独立工具", "workflow_fit": "good", // good | poor | mismatch "workflow_note": "" }
蓝军工具审查 — 创建前的对抗检查
• 功能重叠检查：与现有工具重叠 > 50% → 建议改进而非新建
• 复用概率评估：一次性需求不值得建工具
• 抽象粒度审查：太粗 = 不灵活，太细 = 碎片化
• 判定：approve / revise / block

12蓝军机制 — Blue Agent 对抗审查

Blue Agent 在两个关键节点介入，提供独立的第三方视角，防止 Agent 自我评估的盲区。

检查点 1：工具创建审查
触发：Reflect 判定 need_new_tool = true 时
输入：tool_spec + 现有工具列表 + 任务上下文
审查维度：
• 功能重叠（> 50% → 改进现有工具）
• 复用概率（一次性 vs 通用）
• 抽象粒度（太粗 / 太细）
• 命名和职责清晰度
输出：approve / revise / block
检查点 2：报告质量审查
触发：报告生成完成后
输入：报告全文 + 原始任务描述
审查角度：
• 隐含假设是否被检验？
• 是否存在反例或边界条件？
• 因果链是否可靠？
• 是否过度外推或选择性引用？
输出：结构化挑战列表，追加到报告末尾
蓝军输出示例 (JSON)
[{ "target": "最关注的事 #1", "original_claim": "Claude Code 的 agent 能力已超越 Cursor", "challenge": "该结论基于单一 benchmark，未考虑实际开发场景的多样性", "alternative_view": "不同工具在不同场景各有优势，需多维度对比" }]

13Benchmark 评估体系

量化追踪每次运行的质量指标，让 Agent 知道应该往哪个方向优化。

RunMetrics
metrics.py
每次运行自动采集：
• token 消耗（输入/输出）
• 工具使用情况
• 蓝军挑战/阻止次数
• 报告结构合规性
→ 追加到 metrics.jsonl
ReportChecker
report_checker.py
报告质量检查：
• 标题格式 ✓
• 引用块 ✓
• 来源链接 ✓
• 观察小结 ✓
• URL 有效性 (HTTP HEAD)
ToolHealth
tool_health.py
工具健康度分析：
• 复用率（被使用/总数）
• 零使用工具列表
• 高频工具 Top N
• 多版本工具追踪

14关键设计决策

为什么用独立脚本而非函数库？
• 进程隔离 — 工具崩溃不影响 Agent
• 语言无关 — 未来可支持 Bash/Node
• 可独立测试和调试
• Unix 哲学：stdin/stdout/exit code
为什么用 JSON registry 而非数据库？
• 人类可读，git 友好
• 无额外依赖
• 工具数量级 O(10)，不需要索引
• 与 changelog.md 配合提供完整审计
为什么反思和执行分离？
• 执行关注"做什么" — 调用工具/LLM
• 反思关注"学到什么" — 元认知
• 分离后可独立调优 prompt
• 反思结果可追溯 (changelog)
明确不做的事
• 不做工具沙箱/安全隔离
• 不做自动部署到 Web
• 不做 session 级详细日志
• 不做多 Agent 并行编排

Benchmark ←→ All
North Star Metrics

评测报告

仲裁机制
当 Evolution 和稳定性冲突时裁决
接口示例 — Tool Lifecycle API
# Platform 提供，Evolution 调用 POST /tools/register # 注册新工具 (name, version, script, test_cases) POST /tools/{id}/upgrade # 升级工具 (旧版自动归档) POST /tools/{id}/rollback # 回滚到指定版本 GET /tools/{id}/health # 工具健康度 (Benchmark 数据) GET /traces?run_id=xxx # 执行 trace (Evolution 用于学习)

17张力与平衡 — Evolution 内部的探索节奏

Evolution 内部分工灵活
同一个人可以同时探索 tools 和 workflow
按兴趣和能力自然分化，不需要硬性边界
共同目标：让系统在 Benchmark 上持续进步
Benchmark 的仲裁规则
Health 分数 < 阈值 → 冻结新工具上线
Evolution 分数停滞 → 鼓励更大胆的探索
Accountability 异常 → 强制 review 所有变更
