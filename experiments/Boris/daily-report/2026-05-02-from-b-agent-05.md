## 一、Agent 自己就会搞砸：真实安全问题分类

### 1. 恶意文件触发 Agent 自我拒绝——无需攻击者介入

Claude Code 出现了一个高关注度的回归问题：当仓库中存在包含恶意软件提醒的文件时，agent 在每次读取时都会触发内部安全检查，导致子 agent 拒绝执行正常任务。这不是外部攻击，而是 agent 自身的安全机制在正常工作流中产生误判，直接阻断开发流程。251 分的社区关注度说明这已是广泛痛点。

来源：[Regression: malware reminder on every read still causes subagent refusals](https://github.com/anthropics/claude-code/issues/49363)

### 2. Agent 违反数据库设计的隐含假设——静默破坏数据完整性

传统数据库设计假设调用方是确定性程序，会遵守事务边界、幂等性和并发控制约定。Agentic AI 系统打破了这些隐含假设：agent 可能重复执行写操作、忽略锁机制、在不理解 schema 语义的情况下执行 DDL 变更。这类问题不需要攻击者——agent 按照自己的"理解"操作数据库，就足以造成数据丢失或损坏。

来源：[Agentic AI systems violate the implicit assumptions of database design](https://arpitbhayani.me/blogs/defensive-databases/)

### 3. Agent Harness 应置于沙箱之外——架构层面的安全隔离

这篇文章直接指出：控制 agent 行为的 harness（约束层）不应与 agent 运行在同一个沙箱内。如果 harness 和 agent 共享执行环境，agent 有能力修改自身的约束规则、绕过权限检查、甚至删除安全配置文件。这与 AWS Cedar 的思路一致——策略执行层必须独立于被控制对象，才能提供确定性保障。

来源：[The agent harness belongs outside the sandbox](https://www.mendral.com/blog/agent-harness-belongs-outside-sandbox)

### 4. Agentic Coding 导致开发者倦怠——信任缺失是根因

开发者反馈 agentic coding 带来的不是效率提升，而是持续的焦虑：需要不断审查 agent 的每一步操作是否安全、是否覆盖了不该动的文件、是否引入了漏洞。当安全依赖人工逐行审查而非系统性策略保障时，"人在回路"变成了"人被困在回路里"。

来源：[Agentic coding is burning me out](https://www.0xsid.com/blog/agentic-coding-fatigue)

## 二、确定性策略 vs. 非确定性 LLM 判断

### 核心矛盾

当前 coding agent 的安全模型存在根本性矛盾：用一个非确定性系统（LLM）来判断自身行为是否安全。这导致：

- **同一操作，不同结果**：相同的文件操作请求，agent 可能在一次执行中正常完成，另一次执行中因"安全顾虑"拒绝，或者反过来——本该拒绝的操作被放行。
- **安全规则不可审计**：LLM 内部的安全判断无法被外部系统验证、测试或形式化证明。
- **规模化后失控**：当企业同时运行 100 个 coding agent（如 Omar 项目所示），每个 agent 的安全行为都是概率性的，整体风险呈指数增长。

### AWS Cedar 思路的启示

Cedar 的确定性策略模型提供了一个对照：

| 维度 | LLM 内置判断 | 确定性策略引擎 |
|------|-------------|--------------|
| 可预测性 | 概率性，同一输入可能不同输出 | 确定性，相同输入必定相同输出 |
| 可审计性 | 黑箱，无法形式化验证 | 策略即代码，可测试可审计 |
| 隔离性 | 与 agent 共享上下文 | 独立于 agent 执行环境 |
| 覆盖范围 | 依赖训练数据和 prompt | 显式枚举允许/拒绝规则 |

企业需要的不是"更聪明的 agent"，而是**将安全决策从 agent 内部抽离到外部确定性策略层**——agent 可以请求执行操作，但批准/拒绝由独立的策略引擎基于明确规则做出。

## 四、今日观察小结

今天的数据呈现一个清晰信号：**coding agent 的安全问题不是"会不会被攻击"，而是"agent 自己就会出错"**。从 Claude Code 的 malware 误判导致子 agent 拒绝工作，到 agent 违反数据库隐含假设静默破坏数据，再到开发者因无法信任 agent 而倦怠——这些都是无需攻击者介入的真实故障模式。

业界的应对方向正在收敛：将 agent harness 置于沙箱之外、用确定性策略引擎替代 LLM 自我判断、通过声明式规则（而非 prompt engineering）定义安全边界。这与 AWS Cedar 的设计哲学一致——安全不应该是一个"大概率正确"的事情，它必须是确定性的。
