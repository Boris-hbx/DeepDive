## 一、背景：为什么现在要在 Agent/Harness 层做安全

当前社区正在从"人写代码、AI 辅助"快速切换到"Agent 自主执行、人做审查"的范式。本次采集的 15 条信号中，至少涉及 **Agent Harness 框架**（Pu.sh、Dirac、Omar）、**桌面/终端自动化**（agent-desktop、Loopsy）、**跨机器 Agent 通信**、**大规模 Agent 调度**等多个层面。与此同时，安全问题已经在真实场景中暴露：

- Claude Code 的恶意软件提醒机制导致子 Agent 拒绝执行（251 分高热度），说明 **Agent 内部信任链与策略传播** 已经是生产级痛点。
- Agent 自主发现 GKE WireGuard 网络 Bug，意味着 Agent 具备了探测基础设施漏洞的能力——这把双刃剑的安全面需要被认真对待。
- 400 行 Shell 就能搭出一个完整 coding-agent harness（Pu.sh，89 分），说明 **harness 层的攻击面极低门槛、极易扩散**。

这些信号共同指向一个结论：**安全团队现在介入 Agent/Harness 层，时机恰好，且有大量未被系统性覆盖的课题空间。**

## 二、端到端安全攻击面地图

基于采集数据，将 Agentic AI 的端到端链路拆解为五层，每层标注已观察到的安全风险信号：

| 层次 | 典型组件 | 本次数据中的风险信号 |
|------|----------|---------------------|
| **范式/协议层** | AGENTS.md、Agent 通信协议 | Claude Code 不支持 AGENTS.md，协议碎片化导致策略无法统一下发 |
| **Harness/编排层** | Pu.sh、Dirac、Omar | 400 行 Shell 即可构建 harness，缺乏沙箱与权限隔离 |
| **执行/运行时层** | 桌面自动化、终端操作 | agent-desktop 直接操作原生桌面 GUI，Loopsy 跨机器通信 |
| **模型/推理层** | LLM 本身的安全策略 | 恶意软件提醒导致子 Agent 级联拒绝（策略过杀与策略投毒） |
| **基础设施层** | K8s、网络、存储 | Agent 自主发现 GKE WireGuard Bug，大规模 Agent 服务的扩展性问题 |

## 三、推荐探索课题（按优先级排序）

### 课题 1：Agent 信任链与策略传播安全

**问题**：当前 Agent 系统中，父 Agent 的安全策略如何传递给子 Agent 没有标准机制。Claude Code 的案例（恶意软件提醒在每次读取时触发，导致子 Agent 级联拒绝）暴露了策略传播的两个极端——过杀（合法操作被拒绝）和投毒（攻击者伪造策略让 Agent 放行恶意操作）。

**探索方向**：
- 设计 Agent 间策略传播的信任模型（类似 mTLS 但面向 Agent 语义层）
- 研究"策略注入攻击"：恶意文件内容能否篡改 Agent 的安全判断
- 构建策略一致性验证工具，确保子 Agent 不会因策略冲突而行为异常

**关联信号**：[Regression: malware reminder on every read still causes subagent refusals](https://github.com/anthropics/claude-code/issues/49363)（251 分）、[Claude Code still doesn't support AGENTS.md](https://github.com/anthropics/claude-code/issues/6235)

---

### 课题 2：Harness 层沙箱与权限最小化

**问题**：Pu.sh 用 400 行 Shell 实现了完整的 coding-agent harness，Omar 可以同时管理 100 个 coding agent。这意味着 harness 层正在快速普及，但几乎没有内置的沙箱隔离和权限控制。一个被劫持的 Agent 可以通过 harness 获得宿主机的完整权限。

**探索方向**：
- 为主流 harness 框架设计轻量级沙箱方案（基于 seccomp/landlock/容器）
- 定义 Agent 操作的最小权限模型（文件系统、网络、进程）
- 构建 harness 层的安全基线检测工具（类似 CIS Benchmark 但面向 Agent harness）

**关联信号**：[Pu.sh – a full coding-agent harness in 400 lines of shell](https://pu.dev/)（89 分）、[Omar – A TUI for managing 100 coding agents](https://omar.tech)、[Agentic Harness Engineering](https://arxiv.org/abs/2604.25850)

---

### 课题 3：Agent 自主操作的审计与可观测性

**问题**：Agent 正在获得越来越强的自主操作能力——操作桌面 GUI、跨机器通信、自主调试基础设施。但目前缺乏统一的审计日志标准和实时可观测性方案，出了安全事件很难溯源。

**探索方向**：
- 设计 Agent 操作的结构化审计日志格式（who/what/when/why/with-what-permission）
- 构建 Agent 行为的异常检测系统（基于操作序列的偏离度）
- 研究"Agent 操作回放"能力，支持安全事件的事后取证

**关联信号**：[Agent-desktop – Native desktop automation CLI for AI agents](https://github.com/lahfir/agent-desktop)（88 分）、[Loopsy, a way for terminals and AI agents on different machines to talk](https://github.com/leox255/loopsy)

---

### 课题 4：Agent 间通信的认证与加密

**问题**：Loopsy 这类工具让不同机器上的 Agent 可以互相通信，但 Agent 间通信的认证和加密目前基本是空白。这打开了中间人攻击、Agent 冒充、指令注入等攻击面。

**探索方向**：
- 设计 Agent-to-Agent 的身份认证方案（Agent Identity）
- 研究 Agent 通信中的 Prompt Injection 风险（一个 Agent 通过通信内容操控另一个 Agent）
- 构建 Agent 通信的流量分析与入侵检测原型

**关联信号**：[Loopsy, a way for terminals and AI agents on different machines to talk](https://github.com/leox255/loopsy)（53 分）

---

### 课题 5：Agent 作为攻击工具的红蓝对抗

**问题**：Lovable 的 Agent 自主发现了 GKE 中 WireGuard 的网络 Bug。这证明 Agent 已经具备了自主漏洞发现能力。安全团队应该主动研究 Agent 作为攻击工具的能力边界，以便建立防御。

**探索方向**：
- 构建"红队 Agent"：让 Agent 自主对目标系统进行渗透测试
- 研究 Agent 在漏洞发现、利用链构造方面的能力上限
- 建立 Agent 攻击行为的特征库，用于蓝队检测

**关联信号**：[Our agent found a bug with WireGuard in Google Kubernetes Engine](https://lovable.dev/blog/hunting-networking-bugs-in-kubernetes)（66 分）

---

### 课题 6：大规模 Agent 服务的安全运维

**问题**：智谱分享了 GLM-5 大规模 Agent 服务的扩展痛点。当 Agent 从单实例变成大规模集群服务时，安全运维面临新挑战：Agent 实例间的隔离、密钥管理、异常实例的快速熔断等。

**探索方向**：
- 研究大规模 Agent 集群的安全隔离架构
- 设计 Agent 实例级别的密钥轮转与最小权限凭证分发
- 构建 Agent 集群的安全健康度监控面板

**关联信号**：[Scaling Pain of Coding Agent Serving: Lessons from Debugging GLM-5 at Scale](https://z.ai/blog/scaling-pain)

## 五、给安全团队的落地建议

1. **第一周**：从课题 1 和课题 2 入手。让安全团队先复现 Claude Code 的子 Agent 策略拒绝问题，理解 Agent 信任链的现状；同时对你们现有工程中使用的 harness 做一次攻击面审计。
2. **第二周起**：启动课题 3 和课题 4，为你们的 Agent 系统设计审计日志规范，并对 Agent 间通信做威胁建模。
3. **持续进行**：课题 5 作为长期红蓝对抗能力建设，可以每个迭代投入固定时间。
4. **一个关键原则**：安全团队不应该只做"防守"，而应该深度参与 Agent/Harness 的架构设计。从这批数据看，安全问题往往是架构问题——策略传播是架构问题，权限隔离是架构问题，审计是架构问题。安全团队越早介入架构层，后续的修补成本越低。

## 六、补充观察

- **Agentic 疲劳是真实的**：社区已经出现"Agentic coding is burning me out"的声音，说明当前 Agent 工具链的人机交互设计还不成熟。安全团队在设计安全机制时要注意**不要增加开发者的认知负担**，否则安全机制会被绕过或关闭。
  来源：[Agentic coding is burning me out](https://www.0xsid.com/blog/agentic-coding-fatigue)

- **Karpathy 提出的"Agentic Engineering"概念**值得关注，它代表了从 vibe coding 到工程化的范式转变，安全应该是这个工程化过程中的一等公民。
  来源：[Andrej Karpathy: From Vibe Coding to Agentic Engineering](https://www.youtube.com/watch?v=96jN2OCOfLs)
