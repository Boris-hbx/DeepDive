最关注的事

1. Introducing GPT-5.5

OpenAI 发布新一代旗舰模型 GPT-5.5，官方将 coding、research 和跨工具数据分析列为核心能力方向，同时声称在速度和综合能力上均有提升。对 agent 工程而言，底座模型把「跨工具协作」作为一级能力打磨，意味着原先由应用层 prompt 编排和工具链拼装补齐的能力 gap 正在被模型层吸收，agent 框架的分工边界需要重新审视。

来源：Introducing GPT-5.5

延伸思考

关键要点

  • coding 被提升为模型一级能力而非附属，属底座代际换代而非增量升级
  • 官方强调「across tools」，指向模型层原生多工具调度与上下文整合能力增强
  • 速度与能力同步提升，暗示推理效率架构有实质改进

对我们的启示

  • 应用层自建的 prompt 链与工具编排可能被模型层部分替代，3-6 个月内需重新评估自有 agent 框架的定位与差异化价值
  • LLM 抽象层「可换 backend」的架构设计被再次验证为必要决策；API 开放后应尽快将 GPT-5.5 加入候选 backend，在多步工具调用场景做 A/B 对比
  • 跨工具能力原生化可能改变 MCP 等工具协议层的角色——从「补齐模型短板」转向「提供标准化接口」，协议层设计需关注与模型原生能力的协同而非重叠

2. GPT-5.5 System Card

OpenAI 公开 GPT-5.5 的 System Card，按惯例该文档包含模型在各类 benchmark 上的能力评估、安全红队测试结果及风险缓解措施。对 agent 从业者而言，重点关注其中 agentic 任务（多步工具调用、computer use 等）的评估方法与得分，以及 OpenAI 对模型自主行为设定的安全边界——这些数据直接影响在生产环境中使用 GPT-5.5 作为 agent 底座时的能力预期与风险建模。

来源：GPT-5.5 System Card

延伸思考

关键要点

  • System Card 是 OpenAI 对模型能力与风险的官方基线文档，数据可用于横向对比竞品模型
  • 重点关注 agentic 相关评估维度：多步推理、工具调用、自主决策的得分与失败模式
  • 安全评估部分界定了模型自主行为的边界，影响 agent 系统的 guardrail 设计

对我们的启示

  • 从 System Card 中提取 agentic benchmark 数据，与自有评估体系对齐，建立 GPT-5.5 在多步工具调用场景下的能力基线
  • 关注 OpenAI 对模型自主行为的安全限制描述，评估是否影响现有 agent 架构中的权限设计与人机交互流程
  • 将 System Card 披露的失败模式纳入 agent 异常处理策略，尤其是长链路任务中的错误累积场景

3. A pelican for GPT-5.5 via the semi-official Codex backdoor API

OpenAI 发布 GPT-5.5，面向 Codex 和付费 ChatGPT 用户开放，但官方 API 尚未上线，理由是「API 部署需要不同的安全保障」。Simon Willison 通过 OpenClaw 等 agent 工具借 Codex 半官方接口提前测试，评价为快速高效。这一现象折射出 agent 工具与模型提供商围绕 API 访问权限的持续博弈——订阅价远低于 API 价，agent 工具绕道订阅通道调用模型已成为行业摩擦点。

来源：A pelican for GPT-5.5 via the semi-official Codex backdoor API

延伸思考

关键要点

  • GPT-5.5 已发布但官方 API 未同步开放，OpenAI 以安全为由延迟 API 部署
  • OpenClaw 等 agent 工具通过 Codex 半官方接口绕过 API 限制抢先接入新模型
  • 订阅价与 API 价的价差催生 agent 工具「借道订阅通道」的灰色访问模式
  • Simon 实测评价 GPT-5.5 快速且能力强，但尚无系统性 benchmark 数据

对我们的启示

  • 新模型发布与 API 开放的时间差正在成为常态，agent 框架需设计多通道接入能力以应对节奏错位
  • agent 工具绕道订阅通道调用模型存在政策风险，依赖此路径的工程方案需备降级预案
  • GPT-5.5 API 正式开放后应尽快纳入 backend 候选，重点评估多步工具调用场景表现
  • 模型提供商对 API 访问管控趋严，自建 agent 基础设施时需将访问合规性纳入架构考量
